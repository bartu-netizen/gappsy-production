THE BUG IN 30 SECONDS
================================================================================

WHAT:   Emails in ListClean "clean" bucket with missing result.status were
        being stored as validation_status='unknown' instead of 'valid'

WHY:    Original code: result.status || "unknown" defaulted to unknown when
        status field was missing/empty

IMPACT: • Clean emails excluded from Smartlead
        • Unnecessary retries (6x)
        • Wrong validation status in DB

THE FIX: Use bucket type as fallback when status is missing

4-PART FIX BREAKDOWN
================================================================================

1. BUCKET ANNOTATION (Lines 52-58)
   Preserve source bucket when combining clean/dirty/unknown results
   → Each result carries _bucket: "clean|dirty|unknown"

2. FALLBACK LOGIC (Lines 69-77)
   If result.status missing/empty, use result._bucket as fallback
   → externalStatus = result.status || result._bucket || "unknown"

3. DATABASE AUDIT (Line 116)
   Store bucket source in DB for visibility
   → verification_bucket_source = result._bucket

4. LOGGING (Lines 143-146)
   Log metrics showing how many rows used fallback
   → "[listclean-queue] Batch: XXX rows had missing status..."

ISSUE 1: CORS Preflight Failure (Primary Cause)
──────────────────────────────────────────────
Symptom: "Failed to fetch" error

Flow:
  1. Frontend sends OPTIONS preflight with x-admin-token header
  2. Edge function responds with incomplete CORS headers
  3. CORS headers don't include x-admin-token
  4. Browser rejects preflight
  5. Actual GET request never sent
  6. User sees "Failed to fetch"

Root Cause:
  - Local corsHeaders object missing x-admin-token
  - Imported CORS_HEADERS has it, but not used
  - Inconsistent CORS configuration

Fix:
  - Deleted local corsHeaders definition
  - Use CORS_HEADERS from _shared/adminSession.ts everywhere
  - CORS_HEADERS includes x-admin-token and other required headers

Result:
  - Preflight succeeds ✓
  - GET request sent ✓
  - Download proceeds ✓


ISSUE 2: Scope Error in Catch Block
────────────────────────────────────
Symptom: If export failed, catch block would throw ReferenceError

Code:
  try {
    const responseCorsHeaders = { ... }  // Only defined in try
    // ... logic ...
    return new Response(..., { headers: { ...responseCorsHeaders } });
  } catch (error) {
    return new Response(..., {
      headers: { ...responseCorsHeaders }  // ERROR: not in scope
    });
  }

Problem:
  - responseCorsHeaders only defined inside try block
  - catch block can't access it
  - Error handling would fail with ReferenceError

Fix:
  - Remove responseCorsHeaders declaration
  - Use CORS_HEADERS (module-level import) in all responses
  - CORS_HEADERS is globally scoped, always accessible

Result:
  - Error responses properly formatted ✓
  - CORS headers included in all responses ✓
  - No ReferenceError in error paths ✓


ISSUE 3: URL Revoked Too Early
───────────────────────────────
Symptom: Download may fail or not start

Code:
  a.click();                         // Trigger download
  window.URL.revokeObjectURL(url);  // Immediately revoke
  document.body.removeChild(a);     // Remove anchor

Problem:
  - a.click() queues download, doesn't wait
  - URL revoked before download actually starts
  - Browser can't download from invalidated URL
  - Download may fail silently

Fix:
  - a.click()                    // Trigger download
  - a.remove()                   // Remove from DOM
  - setTimeout(() => {           // Delay revocation
      window.URL.revokeObjectURL(url);
    }, 5000);

Result:
  - Download starts before revocation ✓
  - 5-second safety margin ✓
  - Reliable downloads ✓
